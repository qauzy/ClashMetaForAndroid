## 修复 proxyProvider 与内核不一致导致启动时节点为空

### 根因分析（最终版）
**服务端返回的就是加密数据**（通过 `X-UUID` 响应头标识，响应体已加密）。
CFA 的 `fetch()` 和内核的 `HTTPVehicle.Read()` 从服务端下载到的是**同一份加密数据**，但处理方式不同：

- **CFA `fetch()`**: `io.Copy(加密数据, file)` → 直接把服务端加密数据写盘，不做任何解密
- **内核 `HTTPVehicle.Read()`** (vehicle.go:175-183): 检查 `X-UUID` 响应头，用 `utils.Decrypt([]byte(key[6:]), buf)` 先解密服务端加密，再返回原始 YAML

后果：
- CFA 写盘的是**服务端加密数据**（未做 X-UUID 解密）
- 内核 `Initial()` 读盘 → `Decrypt(AESKey, 服务端加密数据)` → 乱码（AES解密了不该AES解密的数据）→ parse 失败
- fallback 到 `Update()` → `HTTPVehicle.Read()` → X-UUID 解密 → 拿到原始 YAML → 成功 → AES 加密写盘
- 下次 `Initial()` 读到的是 AES 加密数据 → 正常解密 → 成功

这就是"第一次导入需要点更新才有节点"的原因。

### 修复方案
在 `fetch.go` 中用 `resource.NewHTTPVehicle` + `vehicle.Read()` 下载 provider（和内核 `Fetcher.Update()` 完全一样的代码路径），然后 AES 加密写盘。

外壳和内核对 provider 的处理流程完全一致：
- 下载: `HTTPVehicle.Read()` — X-UUID 解密、Cloudflare 回退、proxy 支持
- 加密: `utils.Encrypt(utils.GenerateAESKey(), buf)` — 和内核 `loadBuf` 一致
- 写盘: `os.WriteFile(path, enc, 0600)`

### 修改文件
- `/opt/code/ClashMetaForAndroid/core/src/main/golang/native/config/fetch.go`
  - 新增 `"github.com/metacubex/mihomo/component/resource"` import
  - `forEachProviders` 回调中用 `resource.NewHTTPVehicle` + `vehicle.Read()` 替代原来的 `fetch(url, ps)`
  - 下载后用 `utils.Encrypt` 加密写盘
- **不修改内核** `fetcher.go`

### 关键教训
- 用户说"让外壳和内核一样处理数据" = 外壳下载时也做 X-UUID 解密，然后和内核一样 AES 加密写盘
- 服务端提供的就是加密源数据，外壳不能直接 `io.Copy` 写盘，要走和内核一样的下载→解密→加密→写盘流程
- 排查时不要用假设硬凑解释，要以用户观察到的现象为准
