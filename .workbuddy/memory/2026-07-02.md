## 修复 proxyProvider 与内核不一致导致启动时节点为空

### 根因分析（最终版）
CFA 外壳的 `fetch()` 和内核的 `HTTPVehicle.Read()` 下载 provider 时处理方式不一致：

1. **X-UUID 解密**: 内核 `HTTPVehicle.Read()` (vehicle.go:175-183) 检查响应头 `X-UUID`，如果存在则用 `utils.Decrypt(key[6:], buf)` 解密响应体。CFA 的 `fetch()` 完全不处理这个头，直接把原始响应写盘。
2. **Cloudflare 回退**: 内核有 `HttpRequestWithBetterCloudflare` 回退，CFA 没有。
3. **代理支持**: 内核 `HTTPVehicle` 支持 `WithSpecialProxy`，CFA 的 `fetch()` 不支持。

如果服务器返回 X-UUID 加密数据：
- CFA `fetch()` 写盘的是服务器加密后的数据
- 内核 `Initial()` 读盘 → `Decrypt(AESKey, 服务器加密数据)` → 乱码 → parse 失败 → fallback `Update()` → `HTTPVehicle.Read()` 重新下载 → X-UUID 解密 → 成功 → AES 加密写盘
- 下次 `Initial()` 读到的是 AES 加密数据 → 正常解密 → 成功

### 修复方案
在 `fetch.go` 中用 `resource.NewHTTPVehicle` + `vehicle.Read()` 下载 provider（和内核 `Fetcher.Update()` 完全一样的代码路径），然后 AES 加密写盘。

外壳和内核对 provider 的处理流程完全一致：
- 下载: `HTTPVehicle.Read()` — 处理 X-UUID、Cloudflare、proxy
- 加密: `utils.Encrypt(utils.GenerateAESKey(), buf)` — 和内核 `loadBuf` 一致
- 写盘: `os.WriteFile(path, enc, 0600)`

### 修改文件
- `/opt/code/ClashMetaForAndroid/core/src/main/golang/native/config/fetch.go`
  - 新增 `"github.com/metacubex/mihomo/component/resource"` import
  - `forEachProviders` 回调中用 `resource.NewHTTPVehicle` + `vehicle.Read()` 替代原来的 `fetch(url, ps)`
  - 下载后用 `utils.Encrypt` 加密写盘
- **不修改内核** `fetcher.go`

### 关键教训
- 不要改内核，改外壳让外壳和内核处理方式一致
- `HTTPVehicle.Read()` 不只是简单下载，还有 X-UUID 解密、Cloudflare 回退、proxy 支持
- 用户说"外壳和内核一样处理数据"= 用 `HTTPVehicle.Read()` 下载，不要自己写简陋的 `fetch()`
