## 修复 proxyProvider 与内核不一致导致启动时节点为空

### 根因分析
- CFA `FetchAndValid` 下载 provider 文件保存为未加密的原始 YAML
- mihomo 内核 `Fetcher.Initial()` 读取本地文件时先 `Decrypt(key, buf)`，对未加密数据产生乱码
- Decrypt 对未加密数据的 `PKCS7UnPadding` 可能 panic 导致崩溃
- 即使不 panic，乱码通过 `parser` 解析失败 → fallback 远程 `f.Update()` → 启动时网络不通 → provider 为空 → 节点列表为空

### 修复内容（最终方案）
1. **fetch.go (CFA外壳)**: `FetchAndValid` 下载 provider 文件后，用 `utils.Encrypt(utils.GenerateAESKey(), rawContent)` 加密再写回磁盘，与内核 `loadBuf` 的加密方式完全一致
2. **移除 `os.Stat(ps)` 跳过逻辑**: 原逻辑在文件已存在时跳过下载和加密，导致旧未加密文件永远不会被加密。改为每次都重新下载+加密，确保所有 provider 文件始终是加密的
3. 下载错误不再静默忽略，改为 `log.Warnln` 记录
4. **不修改内核 fetcher.go** — 外壳和内核对 provider 文件的处理方式统一后，内核 Decrypt 自然能正常工作
